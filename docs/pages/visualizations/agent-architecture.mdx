# Agent Architecture Visualization

An interactive network diagram showing how agentful's agents communicate, delegate work, and coordinate development tasks.

## Overview

The Agent Architecture visualization reveals the heart of agentful's autonomous development system: a central **orchestrator** that intelligently delegates work to eight specialized agents, each with unique skills and responsibilities.

This visualization helps you understand:
- How agents communicate with each other
- Which agent handles which types of tasks
- The flow of information through the system
- Dependencies between different development phases

## The Eight Core Agents

### üéØ Orchestrator (Center)

**Role:** Central coordinator that manages all other agents

**Responsibilities:**
- Analyzes product specification and breaks down features
- Delegates tasks to appropriate specialized agents
- Tracks progress across all development phases
- Makes high-level decisions about workflow
- Manages human-in-the-loop checkpoints

**When it works:** Throughout entire development lifecycle

**Delegates to:** All agents based on current phase and task type

---

### üìä Product Analyzer

**Role:** Analyzes product specifications for completeness and clarity

**Responsibilities:**
- Parse `.claude/product/index.md` and domain specifications
- Detect ambiguities, gaps, and unclear requirements
- Calculate product readiness score (0-100%)
- Generate recommendations for improvement
- Identify missing acceptance criteria

**Skills:**
- Requirements analysis
- Gap detection
- Readiness scoring

**When it works:** Before development begins, and when requirements change

**Reports to:** Orchestrator

**Responsibilities:**
- Auto-detect tech stack from package.json, imports, and file structure
- Design system architecture and component boundaries
- Generate domain-specific agents for detected patterns
- Establish coding conventions and patterns
- Make technology recommendations

**Skills:**
- Tech stack detection
- Agent generation
- Pattern analysis

**When it works:** At project start and when new domains are added

**Reports to:** Orchestrator

**Responsibilities:**
- Build REST/GraphQL APIs
- Design and implement database schemas
- Implement authentication and authorization
- Create background jobs and workers
- Handle file uploads and processing

**Skills:**
- API development
- Database design
- Authentication

**When it works:** During implementation phase for backend features

**Reports to:** Orchestrator

**Responsibilities:**
- Create reusable UI components
- Implement pages and views
- Manage client-side state
- Handle forms and validation
- Implement responsive design
- Optimize performance (code splitting, lazy loading)

**Skills:**
- Component development
- State management
- Styling

**When it works:** During implementation phase for frontend features

**Reports to:** Orchestrator

---

### üß™ Tester

**Role:** Writes comprehensive tests

**Responsibilities:**
- Write unit tests for functions and components
- Write integration tests for APIs and workflows
- Write E2E tests for critical user journeys
- Achieve minimum 80% code coverage
- Test edge cases and error scenarios

**Skills:**
- Unit tests
- Integration tests
- E2E tests

**When it works:** After implementation, before quality gates

**Reports to:** Orchestrator

---

### üîç Reviewer

**Role:** Validates code quality and runs automated checks

**Responsibilities:**
- Run six automated quality gates
- Review code for patterns and best practices
- Check documentation completeness
- Validate production readiness
- Generate validation reports

**Skills:**
- Quality checks
- Dead code detection
- Production readiness

**Quality Gates:**
1. Type checking (TypeScript/Flow)
2. Linting (ESLint/Prettier)
3. Tests (all passing)
4. Coverage (minimum 80%)
5. Security (no vulnerabilities)
6. Dead code (no unused exports)

**When it works:** Before review and deployment phases

**Reports to:** Orchestrator

---

### üîß Fixer

**Role:** Automatically remediates validation failures

**Responsibilities:**
- Fix type errors by adding annotations
- Resolve linting issues (formatting, unused vars)
- Fix failing tests or add missing test cases
- Update vulnerable dependencies
- Remove dead code and unused exports

**Skills:**
- Auto-remediation
- Test fixes
- Type error resolution

**When it works:** When quality gates fail

**Reports to:** Orchestrator (receives failures from Reviewer)

## Communication Patterns

### Delegation Flow

The orchestrator uses a hub-and-spoke pattern:

```
Orchestrator ‚Üí Product Analyzer ‚Üí Analyze spec
            ‚Üí Architect ‚Üí Design system
            ‚Üí Backend ‚Üí Implement APIs
            ‚Üí Frontend ‚Üí Build UI
            ‚Üí Tester ‚Üí Write tests
            ‚Üí Reviewer ‚Üí Validate quality
            ‚Üí Fixer ‚Üí Fix issues
```

### Information Flow

Agents share information through structured data files:

- **`.agentful/state.json`** - Current phase, progress, blocking issues
- **`.agentful/architecture.json`** - Detected tech stack, generated agents
- **`.agentful/completion.json`** - Feature completion percentages
- **`.agentful/decisions.json`** - Pending and resolved decisions
- **`.agentful/last-validation.json`** - Quality gate results

### Feedback Loops

**Auto-fix Loop:**
```
Reviewer (fails) ‚Üí Orchestrator ‚Üí Fixer ‚Üí Implementation ‚Üí Reviewer (retry)
```

**Human Checkpoint:**
```
Orchestrator ‚Üí Decision needed ‚Üí Pause ‚Üí Human input ‚Üí Resume
```

## Customization

### Domain-Specific Agents

The **Architect** agent can generate custom agents for your tech stack:

```typescript
// Example: E-commerce domain agent
{
  name: "ecommerce-specialist",
  role: "Implements shopping cart, checkout, payment integration",
  skills: ["Stripe API", "Cart state", "Order processing"],
  triggers: ["payment", "checkout", "cart", "order"]
}
```

These appear as additional nodes connected to the orchestrator.

### Agent Configuration

Customize agent behavior in `.claude/agents/`:

```markdown
# Modify backend agent
You are the Backend Agent with additional expertise in:
- AWS Lambda deployment
- DynamoDB data modeling
- EventBridge event-driven patterns
```

## Best Practices

### When to Use Which Agent

**Use Product Analyzer when:**
- Starting a new project
- Requirements are unclear
- Spec needs validation

**Use Architect when:**
- Tech stack changes
- New domain added
- Architecture needs redesign

**Use Backend when:**
- Building APIs
- Designing database schemas
- Implementing authentication

**Use Frontend when:**
- Creating UI components
- Building pages
- Managing state

**Use Tester when:**
- After implementation
- Adding test coverage
- Testing edge cases

**Use Reviewer when:**
- Before deployment
- Quality validation needed
- Checking production readiness

**Use Fixer when:**
- Quality gates fail
- Auto-remediation possible
- Quick fixes needed
